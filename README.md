# Сопоставление холдингов с условными ГСЗ

Проект для Excel 2016 (русский интерфейс): автоматическое сопоставление названий холдингов из смарт-таблицы `_HOLD_OD` со справочником групп застройщиков `_base_gsz` по гибким текстовым ключам.

## Задача

В рабочей книге Excel есть:

| Таблица | Назначение |
|---------|------------|
| `_HOLD_OD` | Холдинги и показатели ОД; колонка **«Холдинг»** — текст для поиска |
| `_base_gsz` | Справочник ГСЗ: **«Наименование, регион»** и 12 колонок ключей |
| `_UL_HOLD_OD` | Не используется в текущей версии |

Power Query-запрос `_HOLD_OD_GSZ` для каждой строки `_HOLD_OD` ищет совпадения в «Холдинг» по ключам из `_base_gsz` и возвращает:

- **«условное ГСЗ»** — первое совпавшее «Наименование, регион» или `"-"`;
- **«Отладка_совпадения_ГСЗ»** — все совпадения через `";"` и перенос строки (`";\n"`) или `"-"`.

Исходная таблица `_HOLD_OD` не изменяется.

## Логика ключей

### Колонки в `_base_gsz`

| Группа | Колонки | Смысл |
|--------|---------|--------|
| AND full | `key_and_full_1` … `_3` | Обязательные подстроки |
| AND not | `key_and_not_1` … `_3` | Обязательные отдельные слова |
| OR full | `key_or_full_1` … `_3` | Альтернативные подстроки |
| OR not | `key_or_not_1` … `_3` | Альтернативные отдельные слова |

В ячейке — **одно слово**; несколько слов — через колонки `_1` / `_2` / `_3`.

### Модификаторы

| Модификатор | Поведение | Пример |
|-------------|-----------|--------|
| `full` | Подстрока в любом месте | `ПИК` → `ПИКНИК`, `ЗАПИКАТЬ` |
| `not` | Отдельное слово (граница — не буква) | `ПИК` в `ПИКНИК` — **нет** |
| `and` | Все непустые AND-поля; позиции **не пересекаются** | `ПИГ` + `ИГЛА` в `ПИГЛА` — **нет** |
| `or` | Хотя бы одно непустое OR-поле (если OR заполнены) | |

### Формула совпадения строки `_base_gsz`

```
совпадение =
  (все непустые AND-поля выполнены, без пересечения позиций)
  И
  (все OR-поля пусты ИЛИ хотя бы одно OR-поле выполнено)
```

Дополнительно:

- регистр не учитывается (`Text.Lower` + `Text.Trim`);
- буквы: кириллица и латиница;
- строка без ключей не даёт совпадения;
- при нескольких совпадениях: в основной колонке — **первое** по порядку в `_base_gsz`, в отладке — **все**.

### Предфильтр (Python + FAST / FAST_V2)

Ускоренные версии не меняют формулу совпадения, но сначала отсекают кандидатов:

| Состав ключей строки | Что индексируется | Кандидат, если… |
|----------------------|-------------------|-----------------|
| Есть AND | Один якорь из AND (длиннейший `not`, иначе `full`) | якорь найден в тексте |
| Только OR | **Все** непустые OR-токены (`full` и `not`) | найден **хотя бы один** |
| Только `non` / без текстовых ключей | без якоря | строка всегда кандидат (далее полная проверка) |

Так не теряются пары латиница/кириллица в OR (например `Seven` + `СЕВЕН`, `Suns` + `САНС`).  
Базовый `_HOLD_OD_GSZ.pq` предфильтр не использует.

## Power Query: подключение `_HOLD_OD_GSZ`

1. Откройте книгу со смарт-таблицами `_HOLD_OD` и `_base_gsz`.
2. **Данные** → **Создать запрос** → **Пустой запрос**.
3. **Расширенный редактор** → вставьте код из:
   - `src/PowerQuery/_HOLD_OD_GSZ.pq` (базовая версия), или
   - `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq` (ускоренная, без `non`), или
   - `src/PowerQuery/_HOLD_OD_GSZ_FAST_V2.pq` (рекомендуется: `non`-ключи + универсальное число колонок).
4. **Готово** → загрузите как подключение или на новый лист.
5. Переименуйте запрос в:
   - **`_HOLD_OD_GSZ`** для базовой версии, или
   - **`_HOLD_OD_GSZ_FAST`** / **`_HOLD_OD_GSZ_FAST_V2`** для ускоренных версий.

## Ускоренная версия `_HOLD_OD_GSZ_FAST`

Файл: `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq`

Логика совпадений AND/OR и правил `full/not` сохранена, но добавлены оптимизации выполнения:

1. Подготовка ключей `_base_gsz` выполняется один раз (нормализация, списки токенов, флаг наличия ключей).
2. Предфильтр кандидатов: при AND — один якорный токен; при **только OR** — **все** OR-токены (достаточно любого), чтобы не терять варианты латиница/кириллица.
3. Полная проверка `fnСтрокаGszСовпала` запускается только по кандидатам, прошедшим якорный фильтр.
4. Поиск `full`-вхождений использует `Text.PositionOf(..., Occurrence.All)`, а не посимвольный перебор стартовых позиций.

Для больших наборов данных ускоренная версия обычно работает заметно быстрее базовой.

## Ускоренная версия V2 `_HOLD_OD_GSZ_FAST_V2`

Файл: `src/PowerQuery/_HOLD_OD_GSZ_FAST_V2.pq`

Отличия от `_HOLD_OD_GSZ_FAST`:

1. Поддержка исключающих ключей `key_and_non_*` и `key_or_non_*` (логика как в Python-матчере).
2. Универсальное число колонок ключей: автоматически подхватываются все заголовки вида `key_<группа>_<N>` из `_base_gsz`.
3. Сортировка колонок по суффиксу `_N` (например, `_1`, `_2`, `_10`).
4. Можно зафиксировать списки вручную через переменные `Колонки_*_явно` в начале запроса (если список не пустой — автоопределение отключается для этой группы).

Логика `non`:

- `and_non` — исключение, если в названии холдинга (без пробелов) найдены **все** non-ключи;
- `or_non` — исключение, если найден **хотя бы один** non-ключ;
- пример: non-ключ `ммб` исключает и `ну пикммб`, и `ну пик ммб`.

## Python-вариант (полный перенос логики + параллелизация)

Файл: `src/gsz_matcher_parallel.py`

Скрипт реализует ту же бизнес-логику сопоставления, что и Power Query:
- `and/or`,
- `full/not`,
- непересечение позиций AND-токенов.

Дополнительно:
- использует предфильтр по якорям (для AND — один токен; для только OR — все альтернативы; для `not` — через `positions_not`);
- строит индекс кандидатов по anchor-not и anchor-full (в воркерах);
- кэширует позиции токенов в рамках одного холдинга, чтобы не пересчитывать их многократно;
- читает Excel в `read_only` (двухпроходно: границы смарт-таблицы, затем данные);
- запускает сопоставление в несколько процессов (`ProcessPoolExecutor`);
- позволяет задать имена таблиц и колонок через аргументы;
- регресс-тест parity: `src/Tests/test_gsz_matcher_parity.py` (fast vs полный перебор на всех холдингах).

### Зависимости

- Python 3.10+
- `openpyxl`

`openpyxl` обычно доступен в стандартной поставке Anaconda; остальное — стандартная библиотека Python.

### Пример запуска

Базовый запуск (параметры берутся из `config.json`, блок `gsz_matcher_parallel`):

```bash
python3 src/gsz_matcher_parallel.py
```

Запуск с переопределением отдельных параметров через CLI:

```bash
python3 src/gsz_matcher_parallel.py \
  --input-xlsx "input/workbook.xlsx" \
  --output-xlsx "output/hold_od_gsz_python.xlsx" \
  --holding-table "_HOLD_OD" \
  --base-table "_base_gsz" \
  --holding-column "Холдинг" \
  --gsz-column "Наименование, регион" \
  --workers 8 \
  --work-batch-size 30
```

- heartbeat (`[heartbeat] ...`) каждые `heartbeat_seconds`, даже если очередная пачка еще не завершилась;
- долю выполненной работы, скорость, ETA и (опционально) текущий холдинг.

Рекомендуемый "живой" профиль для больших объемов:
- `work_batch_size`: `20`
- `progress_every_holdings`: `500`
- `progress_every_base_rows`: `500`
- `heartbeat_seconds`: `10`

### Формат результата

В выходной Excel пишется лист с холдингами и добавленными колонками (имена настраиваются в `output_format`):
- `Бренд` / `условное ГСЗ` — **первое фактическое** найденное значение (или `-`)
- `Отладка_совпадения_Бренд` / `Отладка_совпадения_ГСЗ` — все совпадения через `";\n"` (или `-`)
- `статус` — текстовый статус сопоставления:
  - `найдено соответствие` — одно совпадение по ключам
  - `зафиксированное значение` — совпадение по `key_fix_id`
  - `есть пересечения по ключам` — несколько совпадений
  - `-` — ничего не найдено
- `Кол-во совпадений` — число совпавших условных ГСЗ

Дополнительно в лист `_base_gsz` добавляются:
- `кол-во холдингов` — сколько холдингов матчилось на эту строку;
- `найденный холдинг` — первое значение `[ID]: холдинг` (или `-`);
- `Отладка_найденного_холдинга` — все найденные холдинги;
- `статус` — статус для строки справочника (`найдено соответствие`, `зафиксированное значение`, `есть пересечения по ключам`, `фикс значение не найдено`, `-`);
- `строка ключа` — сцепка непустых ключей через `_`;
- `длина ключа` — число символов в `строке ключа`;
- `число повторов` — сколько раз такая же `строка ключа` встречается в `_base_gsz`.

### Ключ `key_fix_id` (фиксированное соответствие)

Колонка `key_fix_id` на `_base_gsz` задаёт жёсткую привязку строки ГСЗ к холдингу по **ID холдинга** (один ID или несколько через `"; "`).

| Ситуация | Поведение |
|----------|-----------|
| ID найден в `_HOLD_OD` | Фиксированное соответствие без проверки текстовых ключей |
| ID не найден | Fallback на обычный поиск по ключам; если и он пуст — статус `фикс значение не найдено` на `_base_gsz` |
| Пустой `key_fix_id` | Только поиск по ключам |

Имя колонки задаётся в `config.json`: `"fix_id_col": "key_fix_id"`.

Исключающие ключи в `_base_gsz`:
- `key_and_non_*` — исключение с логикой AND: строка исключается только если найдены **все** non-ключи;
- `key_or_non_*` — исключение с логикой OR: строка исключается, если найден **хотя бы один** non-ключ;
- проверка non выполняется по названию холдинга **без пробелов** (например, `ММБ` исключает и `ну пикммб`, и `ну пик ММБ`).

Форматирование выходных листов управляется через `output_format`:
- заголовки по центру (`header_center`)
- перенос строк в заголовках (`header_wrap`)
- жирные заголовки (`header_bold`)
- закрепление на листе `_HOLD_OD` (`holding_freeze_rows`, `holding_freeze_cols`)
- закрепление на листе `_base_gsz` (`base_freeze_rows`, `base_freeze_cols`) — для 1 строки и 3 колонок: `1` и `3` (ячейка `D2`)
- минимальная ширина всех колонок (`min_width_all`)
- общая минимальная ширина всех колонок (`min_width_all`)
- добавляемые колонки задаются по **ключу типа данных** в `holding_sheet.columns` и `base_sheet.columns`:
  - ключ фиксированный (например, `gsz_primary`, `found_holding_debug`)
  - `name` — настраиваемый заголовок
  - `width` — минимальная ширина колонки
  - `wrap` — перенос строк (опционально)
- вертикальное центрирование всех ячеек обоих листов (`format_data_vertical_center`, по умолчанию `true`; `false` ускоряет запись)
- автофильтр включается для каждого листа по всей области данных

Ключи для `holding_sheet.columns`:
- `gsz_primary` — основное условное ГСЗ / Бренд
- `gsz_debug` — отладка совпадений ГСЗ
- `match_status` — колонка «статус»
- `match_count` — количество совпадений

Ключи для `base_sheet.columns`:
- `holding_count` — количество найденных холдингов
- `found_holding` — первый найденный холдинг (`[ID]: холдинг`)
- `found_holding_debug` — отладка найденных холдингов
- `match_status` — колонка «статус»
- `key_string` — строка ключа
- `key_length` — длина строки ключа
- `key_repeat_count` — число повторов строки ключа

### Регресс-тест (parity)

Проверка, что предфильтр (AND — один якорь; только OR — все альтернативы) даёт тот же результат, что и полный перебор:

```bash
python3 -m unittest src.Tests.test_gsz_matcher_parity -v
```

На тестовой книге `input/workbook.xlsx` (26 000 холдингов × 6 921 строк справочника) полный прогон занимает ~7–9 с (зависит от форматирования выходного файла).

## Генератор ключей `_base_gsz`

Скрипт `src/generate_gsz_keys.py` читает смарт-таблицу `_base_gsz` из Excel-книги
(`gsz_matcher_parallel.input_xlsx`, колонка `gsz_column`) и создаёт локальный Excel с ключами.

Состав колонок ключей берётся из `config.json`:
`and_full_cols`, `and_not_cols`, `and_non_cols`, `or_full_cols`, `or_not_cols`, `or_non_cols`.

### Запуск

```bash
pip install openpyxl   # один раз
python3 src/generate_gsz_keys.py
```

Результат: `output/_base_gsz_keys_YYYYMMDD_HHMMSS.xlsx` (если `output_add_timestamp=true`).

### Правила подбора ключей

1. Источник — `input_xlsx` / таблица `_base_gsz` / колонка «Наименование, регион».
2. Берётся часть **до запятой** (`ПИК, г.Москва` → `ПИК`).
3. Дефис разбивает на части: `ВКБ-Новостройки` → `ВКБ` + `Новостройки`; `Строй-А` → `Строй` + `А`.
4. Убираются служебные формы: ООО, ГК, Холдинг и т.д. (не брендовые слова вроде «Строй»).
5. Короткие токены (≤5 букв) → `not`, длинные → `full`.
6. **Ключ только из своего имени** (или транслит RU↔EN того же токена). Чужие/выдуманные слова запрещены.
7. Отдельные буквы (`А` в `Строй-А`) → `or_not` + транслит (`A`).
8. `*_non` автозаполнением не выдумываются (для ручной правки).
9. Если уникальность невозможна — `ключи_задублированы = да`.

### Колонки выходного Excel

| Колонка | Описание |
|---------|----------|
| `Наименование, регион` | Исходная строка (имя из `gsz_column`) |

| `key_and_non_*` / `key_or_non_*` | Исключающие non-ключи |
| `key_or_full_*` / `key_or_not_*` | OR-ключи (если AND не заполнен) |
| `ключи_задублированы` | `да` — не удалось развести однозначно |
| `комментарий_ключей` | Причина |
