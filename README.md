# Сопоставление холдингов с условными ГСЗ

Проект для Excel 2016 (русский интерфейс): автоматическое сопоставление названий холдингов из смарт-таблицы `_HOLD_OD` со справочником групп застройщиков `_base_gsz` по гибким текстовым ключам.

## Задача

В рабочей книге Excel есть:

| Таблица | Назначение |
|---------|------------|
| `_HOLD_OD` | Холдинги и показатели ОД; колонка **«Холдинг»** — текст для поиска |
| `_base_gsz` | Справочник ГСЗ: **«Наименование, регион»** и 12 колонок ключей |
| `_UL_HOLD_OD` | Не используется в текущей версии |

Power Query-запрос `_HOLD_OD_GSZ` для каждой строки `_HOLD_OD` ищет совпадения в «Холдинг» по ключам из `_base_gsz` и возвращает:

- **«условное ГСЗ»** — первое совпавшее «Наименование, регион» или `"-"`;
- **«Отладка_совпадения_ГСЗ»** — все совпадения через `"; "` или `"-"`.

Исходная таблица `_HOLD_OD` не изменяется.

## Логика ключей

### Колонки в `_base_gsz`

| Группа | Колонки | Смысл |
|--------|---------|--------|
| AND full | `key_and_full_1` … `_3` | Обязательные подстроки |
| AND not | `key_and_not_1` … `_3` | Обязательные отдельные слова |
| OR full | `key_or_full_1` … `_3` | Альтернативные подстроки |
| OR not | `key_or_not_1` … `_3` | Альтернативные отдельные слова |

В ячейке — **одно слово**; несколько слов — через колонки `_1` / `_2` / `_3`.

### Модификаторы

| Модификатор | Поведение | Пример |
|-------------|-----------|--------|
| `full` | Подстрока в любом месте | `ПИК` → `ПИКНИК`, `ЗАПИКАТЬ` |
| `not` | Отдельное слово (граница — не буква) | `ПИК` в `ПИКНИК` — **нет** |
| `and` | Все непустые AND-поля; позиции **не пересекаются** | `ПИГ` + `ИГЛА` в `ПИГЛА` — **нет** |
| `or` | Хотя бы одно непустое OR-поле (если OR заполнены) | |

### Формула совпадения строки `_base_gsz`

```
совпадение =
  (все непустые AND-поля выполнены, без пересечения позиций)
  И
  (все OR-поля пусты ИЛИ хотя бы одно OR-поле выполнено)
```

Дополнительно:

- регистр не учитывается (`Text.Lower` + `Text.Trim`);
- буквы: кириллица и латиница;
- строка без ключей не даёт совпадения;
- при нескольких совпадениях: в основной колонке — **первое** по порядку в `_base_gsz`, в отладке — **все**.

## Power Query: подключение `_HOLD_OD_GSZ`

1. Откройте книгу со смарт-таблицами `_HOLD_OD` и `_base_gsz`.
2. **Данные** → **Создать запрос** → **Пустой запрос**.
3. **Расширенный редактор** → вставьте код из:
   - `src/PowerQuery/_HOLD_OD_GSZ.pq` (базовая версия), или
   - `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq` (рекомендуется для больших объемов, например 44k+ холдингов).
4. **Готово** → загрузите как подключение или на новый лист.
5. Переименуйте запрос в:
   - **`_HOLD_OD_GSZ`** для базовой версии, или
   - **`_HOLD_OD_GSZ_FAST`** для ускоренной версии.

## Ускоренная версия `_HOLD_OD_GSZ_FAST`

Файл: `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq`

Логика совпадений AND/OR и правил `full/not` сохранена, но добавлены оптимизации выполнения:

1. Подготовка ключей `_base_gsz` выполняется один раз (нормализация, списки токенов, флаг наличия ключей).
2. Для каждой строки справочника выбирается "якорный" токен для дешевой предварительной фильтрации кандидатов.
3. Полная проверка `fnСтрокаGszСовпала` запускается только по кандидатам, прошедшим якорный фильтр.
4. Поиск `full`-вхождений использует `Text.PositionOf(..., Occurrence.All)`, а не посимвольный перебор стартовых позиций.

Для больших наборов данных ускоренная версия обычно работает заметно быстрее базовой.

## Python-вариант (полный перенос логики + параллелизация)

Файл: `src/gsz_matcher_parallel.py`

Скрипт реализует ту же бизнес-логику сопоставления, что и Power Query:
- `and/or`,
- `full/not`,
- непересечение позиций AND-токенов.

Дополнительно:
- использует предфильтр по "якорному" токену,
- строит индекс кандидатов по anchor-not и anchor-full (в воркерах),
- кэширует позиции токенов в рамках одного холдинга, чтобы не пересчитывать их многократно,
- запускает сопоставление в несколько процессов (`ProcessPoolExecutor`),
- позволяет задать имена таблиц и колонок через аргументы.

### Зависимости

- Python 3.10+
- `openpyxl`

`openpyxl` обычно доступен в стандартной поставке Anaconda; остальное — стандартная библиотека Python.

### Пример запуска

Базовый запуск (параметры берутся из `config.json`, блок `gsz_matcher_parallel`):

```bash
python3 src/gsz_matcher_parallel.py
```

Запуск с переопределением отдельных параметров через CLI:

```bash
python3 src/gsz_matcher_parallel.py \
  --input-xlsx "input/workbook.xlsx" \
  --output-xlsx "output/hold_od_gsz_python.xlsx" \
  --holding-table "_HOLD_OD" \
  --base-table "_base_gsz" \
  --holding-column "Холдинг" \
  --gsz-column "Наименование, регион" \
  --workers 8 \
  --chunk-size 200
```

### Настройка через `config.json`

Скрипт по умолчанию читает `config.json` из корня проекта и ожидает блок:

```json
{
  "gsz_matcher_parallel": {
    "input_xlsx": "input/workbook.xlsx",
    "output_xlsx": "output/hold_od_gsz_python.xlsx",
    "holding_table": "_HOLD_OD",
    "base_table": "_base_gsz",
    "holding_column": "Холдинг",
    "gsz_column": "Наименование, регион",
    "and_full_cols": ["key_and_full_1", "key_and_full_2", "key_and_full_3"],
    "and_not_cols": ["key_and_not_1", "key_and_not_2", "key_and_not_3"],
    "or_full_cols": ["key_or_full_1", "key_or_full_2", "key_or_full_3"],
    "or_not_cols": ["key_or_not_1", "key_or_not_2", "key_or_not_3"],
    "workers": 8,
    "chunk_size": 100,
    "work_batch_size": 20,
    "log_stages": true,
    "progress_every_holdings": 500,
    "progress_every_base_rows": 500,
    "progress_every_batches": 25,
    "progress_every_read_rows": 500,
    "heartbeat_seconds": 10,
    "show_current_holding": true,
    "diagnose_workbook_objects": true,
    "log_to_file": true,
    "logs_dir": "LOGS",
    "log_file_prefix": "gsz_matcher_parallel"
  }
}
```

Во время длинной обработки скрипт выводит:
- стадии выполнения (`[stage] ...`);
- прогресс подготовки справочника (`[progress-base] ...`) каждые `progress_every_base_rows`;
- этапный прогресс по завершению батчей (`[progress-batch] ...`) каждые `progress_every_batches`;
- прогресс чтения и маппинга строк таблиц (`[progress-read] ...`) каждые `progress_every_read_rows`;
- прогресс каждые `progress_every_holdings` холдингов;
- heartbeat (`[heartbeat] ...`) каждые `heartbeat_seconds`, даже если очередная пачка еще не завершилась;
- долю выполненной работы, скорость, ETA и (опционально) текущий холдинг.
- диагностику объектов книги (`[diag] ...`): список Smart Tables и Defined Names, а также проверку, что `holding_table/base_table` найдены именно как Smart Table.
- файловый лог в каталоге `LOGS` с именем вида `gsz_matcher_parallel_YYYYMMDD_HHMMSS.log`.

Чтобы прогресс в сопоставлении появлялся чаще, уменьшайте `work_batch_size` (например, 20-50).

Рекомендуемый "живой" профиль для больших объемов:
- `work_batch_size`: `20`
- `progress_every_holdings`: `500`
- `progress_every_base_rows`: `500`
- `heartbeat_seconds`: `10`

Для альтернативного файла конфигурации:

```bash
python3 src/gsz_matcher_parallel.py --config-json "path/to/config.json"
```

### Что можно настроить

- таблицы: `--holding-table`, `--base-table`
- основные колонки: `--holding-column`, `--gsz-column`
- группы ключей:
  - `--and-full-cols`
  - `--and-not-cols`
  - `--or-full-cols`
  - `--or-not-cols`
- производительность: `--workers`, `--chunk-size`
- загрузка параметров из JSON: `--config-json`

### Формат результата

В выходной Excel пишется лист с холдингами и добавленными колонками:
- `условное ГСЗ` (первое совпадение или `-`)
- `Отладка_совпадения_ГСЗ` (все совпадения или `-`)

### Шаги запроса (основные)

| Шаг | Назначение |
|-----|------------|
| `Источник_HOLD_OD` / `Источник_base_gsz` | Загрузка смарт-таблиц |
| `Справочник_буфер` | Буферизация справочника |
| `fnНормализоватьТекст` | Trim + нижний регистр |
| `fnВсеПозицииFull` / `fnПозицииNot` | Поиск full / not |
| `fnПроверитьAndТокены` | AND без пересечения |
| `fnПроверитьOrБлок` | OR-блок |
| `fnСтрокаGszСовпала` | Проверка одной строки справочника |
| `fnНайтиВсеGsz` | Все совпадения для холдинга |
| `СКолонкой_список_совпадений` | Список совпадений (один проход) |
| `СКолонкой_условное_ГСЗ` / `СКолонкой_отладка` | Итоговые колонки |
| `Без_служебной_колонки` | Удаление `СовпаденияГСЗ` |
| `Результат` | Финальная таблица |

### Совместимость Excel 2016

- Не используются `Table.ExpandRecordColumn` с 4 аргументами и `Table.AddColumn` с `type` — только отдельные шаги `AddColumn`.
- Доступ к колонкам через `Record.Field(_, "имя")`.
- `List.Count = 0` вместо `List.IsEmpty`.

## Генератор ключей `_base_gsz`

Скрипт `src/generate_gsz_keys.py` читает `КейЛОАД.txt` и создаёт Excel локально.

### Запуск

```bash
pip install openpyxl   # один раз
python3 src/generate_gsz_keys.py
```

Результат: `output/_base_gsz_keys.xlsx` (смарт-таблица `_base_gsz`).

### Правила подбора ключей

1. Берётся часть **до запятой** (`ПИК, г.Москва` → `ПИК`).
2. Дефис разбивает на части: `ВКБ-Новостройки` → `ВКБ` + `Новостройки` (не целая строка с дефисом).
3. Убираются служебные слова: ГК, Группа, Холдинг, Фонд и т.д.
4. Короткие токены (≤5 букв) → `not`, длинные → `full`.
5. AND до 3 токенов; проверка уникальности по той же логике, что в PQ.
6. Если уникальность невозможна — одинаковые ключи + `ключи_задублированы = да`.

### Колонки выходного Excel

| Колонка | Описание |
|---------|----------|
| `Наименование, регион` | Исходная строка |
| `key_and_full_*` … `key_or_not_*` | Ключи |
| `ключи_задублированы` | `да` — не удалось развести однозначно |
| `комментарий_ключей` | Причина |

### config.json

| Параметр | По умолчанию | Описание |
|----------|--------------|----------|
| `input_file` | `КейЛОАД.txt` | Входной список |
| `output_file` | `output/_base_gsz_keys.xlsx` | Выходной Excel |
| `min_token_length` | `2` | Минимальная длина токена |
| `short_token_not_max_len` | `5` | Порог для режима `not` |
| `common_token_threshold` | `80` | Зарезервировано |
| `gsz_matcher_parallel.*` | см. блок в примере выше | Настройки Python-матчера (файлы, таблицы, колонки, ключи, производительность) |
