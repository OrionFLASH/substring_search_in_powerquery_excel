## 1. Суть доработки

Добавить в справочник `_base_gsz` новую колонку **`key_fix_id`** — жёсткая привязка строки ГСЗ к конкретному холдингу (или нескольким) по **ID холдинга**.

Если в `key_fix_id` задано значение и соответствующий ID найден в списке холдингов (`_HOLD_OD`), считаем, что найдено **зафиксированное** соответствие «этот ГСЗ ↔ этот холдинг». Именно его подтягиваем в результат.

Если ID из `key_fix_id` **не найден** среди холдингов — к этой строке применяется **стандартный поиск по ключам** (`and_*`, `or_*`, `*_non`).

Дополнительно вводится отдельная колонка **«статус»** (имя задаётся в `config.json`). Тексты вроде «есть пересечения по ключам» **переносятся из основных колонок в статус**; в колонках «Бренд» / «найденный холдинг» и отладки остаются только фактические значения совпадений.

---

## 2. Формат `key_fix_id`

| Правило | Описание |
|---------|----------|
| Расположение | Колонка на листе `_base_gsz` |
| Одно значение | Один ID холдинга (как в колонке `ID холдинга` на `_HOLD_OD`) |
| Несколько значений | Несколько ID через разделитель `"; "` (точка с запятой + пробел) |
| Пустая ячейка | Фиксация не задана → только стандартный поиск по ключам |

**Пример:** `12345` или `12345; 67890`

---

## 3. Логика сопоставления

### 3.1 Приоритет

```
1. Если key_fix_id не пустой:
   a) Разобрать список ID (split по "; ")
   b) Найти холдинги с этими ID в _HOLD_OD
   c) Если найден ровно один холдинг → зафиксированное соответствие
   d) Если найдено несколько ID из списка → уточнить у заказчика (см. п. 6)
   e) Если ни один ID не найден → fallback на стандартный поиск по ключам

2. Если key_fix_id пустой:
   → только стандартный поиск по ключам
```

### 3.2 Зафиксированное соответствие

- Строка `_base_gsz` с заполненным `key_fix_id` и найденным холдингом **не проходит** обычную проверку ключей для этой пары — связь считается установленной явно.
- Обратная проекция: холдинг «видит» этот ГСЗ; строка ГСЗ «видит» этот холдинг.

### 3.3 Fallback при ненайденном fix-ID

| Колонка | Значение |
|---------|----------|
| **Статус** | `фикс значение не найдено` |
| **Бренд** / **найденный холдинг** | Результат **обычного** поиска по ключам (как сейчас, но без подстановки статусных фраз) |
| **Отладка** | Результат **обычного** поиска по ключам |

Если при fallback обычный поиск тоже ничего не дал — статус пересчитывается по общим правилам (см. п. 4), т.е. может стать `"-"`.

## 4. Колонка «статус» — допустимые значения

Имя колонки задаётся в `config.json` (отдельно для листа холдингов и листа `_base_gsz`, если потребуется).

| Ситуация | Текст статуса |
|----------|---------------|
| Одно совпадение по ключам (без fix или после fallback) | `найдено соответствие` |
| Совпадение по `key_fix_id` (ID найден в холдингах) | `зафиксированное значение` |
| Несколько совпадений по ключам | `есть пересечения по ключам` |
| `key_fix_id` задан, но ID не найден среди холдингов | `фикс значение не найдено` |
| Ничего не найдено (ни fix, ни ключи) | `-` |

### 4.1 Что убрать из основных колонок

Сейчас при нескольких совпадениях в колонку «Бренд» / «найденный холдинг» пишется фраза `есть пересечения по ключам`.  
**После доработки:** эта фраза — **только в статусе**. В «Бренд» и отладке — список фактических значений (первое / все через разделитель), как при обычном множественном совпадении.
