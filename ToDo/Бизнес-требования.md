# Бизнес-требования. БлицАндеррайтинг

## 1. Назначение проекта

**БлицАндеррайтинг** — автоматизированный калькулятор для экспресс-анализа сделок девелоперского проектного финансирования жилой недвижимости.

Цель: за минуты (вместо дней) получить предварительное решение по сделке — продукт, лимиты, метрики, риски и рекомендации.

### Что считает калькулятор

- Определяет продукт финансирования (ПФ, бридж, 2в1) по параметрам проекта
- Определяет категорию застройщика и характеристики проекта согласно кредитной стратегии банкаъ
- Рассчитывает метрики проекта: `ltc_value`, `ltarv_value`, `llcr_max`, `equity_value`
- Определяет нормативные значения `ltc_min`, `ltarv_min`, `equity_min` по категории и региону
- Выявляет стоп-факторы, условия (жёсткие требования) и риски
- Формирует предварительное решение (светофор: красный / жёлтый / зелёный)
- Подготавливает описание рисков и меры минимизации через LLM

### Артефакты калькулятора

**Отчёты:**
- **Для КМ ККСБ (внутренний)** — полный отчёт: метрики, условия, риски, меры минимизации, ставки финансирования
- **Для клиента (Pre-TS)** — индикативные условия: ставки, комиссии, обеспечение, требования

### Пользователи

**КМ ККСБ** — Клиентский менеджер Корпоративного клиентского сервиса банка. Роль-пользователь, ведущий сделку и получающий внутренний отчёт.

## 2. Глоссарий

### Термины предметной области

**РнС** — Разрешение на строительство. Документ, подтверждающий соответствие проектной документации требованиям градостроительного плана и дающий право начать строительство.

**ТЭП** — Технико-экономические показатели проекта. Утверждённые параметры: площадь, этажность, количество очередей, состав помещений и т.д. При наличии РнС ТЭП считаются утверждёнными автоматически.

**ИЖС** — Индивидуальное жилищное строительство. Тип недвижимости, для которого применяются отдельные корректировки LLCR.

**Апартаменты** — Тип недвижимости, не являющийся жилым фондом (служебный/нежилой статус). Влияет на корректировку LLCR и применение cap.

**ГК** — Группа компаний. Совокупность юридических лиц под единым управлением, выступающая заёмщиком по сделке.

**Tier (категория ГК)** — Классификация группы компаний по надёжности:
- **Tier 1** — ТОП-1 категория (высшая)
- **Tier 2** — ТОП-2 категория (высокая)
- **Tier 3** — ТОП-не 1 и 2 категории
- **Tier 4** — Не-ТОП застройщик

**Проект в фокусном регионе** — Параметр, определяемый по таблице `developers`: совпадает ли регион проекта с одним из значений в массиве `focus_regions` застройщика. Характеризует отношение застройщика к региону.

**Высокопотенциальный регион** — Параметр, определяемый по таблице `regions` (колонка `is_high_potential`). Характеризует сам регион как перспективный для банка.

**Equity** — Собственные средства заёмщика, вложенные в проект. Рассчитывается как `1 − ltc_value`.

**ARV (After-Repair Value)** — Рыночная стоимость прав на участок после получения РнС. Оценка потенциальной стоимости актива на момент готовности проектной документации.

### Продукты финансирования

**ПФ (Проектное финансирование)** — Кредитный продукт для проектов с действующим РнС. Основной продукт банка в сфере девелоперского финансирования.

**Предпроектный бридж** — Кредитный продукт для проектов без РнС и без утверждённых ТЭП. Цель — финансирование расходов до получения РнС.

**Проектный бридж** — Кредитный продукт для проектов без РнС, но с утверждёнными ТЭП. Переходная ступень между предпроектным бриджем и полным ПФ.

**Продукт 2в1** — Сочетание бриджа (предпроектного или проектного) с зафиксированным лимитом на будущее проектное финансирование после получения РнС.

### Финансовые метрики

> **Соглашение об именах:** `_value` — расчётное значение (по формуле), `_min` — норматив (минимально допустимое), `_max` — норматив (максимально требуемое).

**LTC (Loan-to-Cost)** — Отношение суммы кредита к базе стоимости проекта.

Формула: **`ltc_value` = `credit_for_rights` / база_LTC**

База для LTC зависит от наличия РнС:
- При РнС = НЕТ: `expenses_cost` → `rights_price` → `market_value_rights` (первое непустое)
- При РнС = ДА: `rights_price` → `market_value_rights` (первое непустое)

Норматив: **`ltc_min`** — берётся из таблицы `strategies` (тип «нормативы» или «стратегия»). `ltc_value` не должен превышать `ltc_min`.

**LTARV (Loan-to-ARV)** — Отношение суммы кредита к рыночной стоимости с РнС.

Формула: **`ltarv_value` = `credit_for_rights` / `market_value_rights`**

Норматив: **`ltarv_min`** — берётся из таблицы `strategies`. `ltarv_value` не должен превышать `ltarv_min`.

**LLCR (Loan Life Coverage Ratio)** — Коэффициент покрытия долгового сервиса за срок кредита.

Формула нормативного LLCR:

**`llcr_max` = `llcr_base` + Σ(надбавки из таблицы strategies, тип «надбавка LLCR»)**

Где:
- **`llcr_base`** = 1,20 — базовое значение
- Надбавки включают: категорию ГК, группу риска региона, опыт, характеристики проекта (апартаменты, нет ТЭП, нет РнС, срок > 7 лет, ИЖС), «лучше рынка»
- Положительные и отрицательные значения суммируются

Ограничения для жилой недвижимости (не ИЖС, не апартаменты):
- **CAP** (верхняя граница): `llcr_max` не превышает **1,35**
- **FLOOR** (нижняя граница): `llcr_max` не ниже **1,15** (при РнС = ДА) или **1,20** (при РнС = НЕТ)

Признаки: `llcr_capped` = true/false, `llcr_floored` = true/false. Значение до ограничений: `llcr_uncapped`.

**`llcr_value` (LLCR клиента)** — Фактическое значение LLCR по финансовой модели заёмщика. По умолчанию 1,20. Сравнивается с `llcr_max`: если `llcr_value` ≥ `llcr_max` — требование выполнено.

**Equity** — Собственные средства заёмщика.

Формула: **`equity_value`(%) = (1 − `ltc_value`) × 100%**

Норматив: **`equity_min`** — берётся из таблицы `strategies`. `equity_value` не должен быть ниже `equity_min`.

### Стоп-факторы, условия и риски

**Стоп-факторы** — Категорические условия, при которых сделка отклоняется автоматически (светофор = красный). Это не риск и не условие — это отказ от работы с проектом. Хранятся в таблице `strategies` (тип «стоп фактор»).

**Условия (жёсткие требования)** — Проверки по расчётным метрикам и параметрам. Каждому присваивается «да» или «нет». Влияют на итоговое решение (светофор). Хранятся в таблице `risks` (тип «жесткое требование»):
- `LTC_BREACH` — превышение норматива LTC
- `LTARV_BREACH` — превышение норматива LTARV
- `LLCR_LOW` — LLCR клиента ниже норматива
- `NO_EXPERIENCE` — отсутствие опыта группы (опыт «Отсутствует»)
- `LOW_EXPERIENCE` — недостаточный опыт группы (опыт «Недостаточный»)

> **Примечание:** Показатель Equity (`equity_value` / `equity_min`) рассчитывается и выводится в отчёте, но отдельный флаг `LOW_EQUITY` удалён. Нарушение собственного капитала объясняется через `LTC_BREACH` и `LTARV_BREACH`.

**Риски** — Факторы, требующие обсуждения с клиентом и структурирования. Каждому присваивается «да» или «нет». Хранятся в таблице `risks` (тип «риск»). Выводятся в отчёте с описанием и мерами минимизации.

**Decision (решение)** — Итоговое предварительное решение системы:
- **Отказ от сделки** — сработал стоп-фактор (красный)
- **Одобрено с отклонениями** — есть условия со значением «да» (жёлтый)
- **Одобрено без отклонений** — нет нарушений (зелёный)

### 4.1 Таблица `developers`

**Назначение:** справочник всех застройщиков (юрлиц) с историей ввода жилых площадей по годам.

**Кто заполняет:** администратор. Версионирование не применяется.

| Колонка | Системное имя | Тип | Описание |
|---------|---------------|-----|----------|

## 5. Сценарий вызова калькулятора

### Предусловие

КМ ККСБ работает в карточке потенциального лида. В карточке лида заполнены реквизиты компании-клиента: наименование группы компаний, ИНН, наименование холдинга (если входит в состав холдинга). Продуктовая группа сделки — «Финансирование недвижимости».

### Вызов

В карточке лида отображается блок **«Экспресс-расчёт»**. КМ ККСБ нажимает кнопку **«Перейти к расчёту»**.

Открывается модальное окно калькулятора «БлицАндеррайтинг». Реквизиты компании подтягиваются из карточки лида автоматически.

## 7. Алгоритмы UI

### 7.1 Алгоритм определения продукта

**Шаг 1.** Пользователь выбирает тип сделки:
- Покупка прав (`acquiring_rights` = Y)
- Реализация собственного проекта (`acquiring_rights` = N)

**Шаг 2.** Переключатель «Разрешение на строительство»:
- ДА (`has_rns` = Y) → продукт **«Проектное финансирование»**. Конец.
- НЕТ (`has_rns` = N) → появляются поля шагов 3 и 4

**Шаг 3.** Переключатель «Утверждены ТЭП» (`has_teps`)

**Шаг 4.** Переключатель «Требуется лимит ПФ» (`need_pf_limit`)

**Функция выбора продукта:**

```java
public String resolveProduct(boolean hasRns, boolean hasTeps, boolean needPfLimit) {
    if (hasRns) {
        return "Проектное финансирование";
    }
    String bridge = hasTeps ? "Проектный бридж" : "Предпроектный бридж";
    if (needPfLimit) {
        return "Продукт 2в1";
    }
    return bridge;
}
```

Флажок `acquiring_rights` на выбор продукта не влияет — он определяет структуру сделки (появление сумм покупки прав), но не сам продукт.

### 7.2 Алгоритм появления финансовых полей

Могут появляться три финансовых поля:
- **Стоимость расходов до РнС** — `expenses_cost` (при РнС = НЕТ) или `rights_price` (при РнС = ДА)
- **Сумма бриджа** — `credit_for_rights`
- **Рыночная стоимость участка с РнС** — `market_value_rights`

**Шаг 1. Нужны ли финансовые поля вообще?**

Финансовые поля появляются, если выполняется хотя бы одно:
- Тип сделки — «Покупка прав» (`acquiring_rights` = Y)
- РнС = НЕТ (`has_rns` = N)

Если «Реализация собственного проекта» **И** РнС = ДА — поля **не появляются**. Подсказка: «Это чистое проектное финансирование: заёмщик реализует собственный проект с действующим РнС.»

**Шаг 2. Какие поля появляются и с какими подсказками?**

Если поля нужны — видны все три. Смысл первого поля зависит от РнС.

#### Сценарий А: Покупка прав + РнС = НЕТ

| Поле | Системное имя | Подсказка |
|------|---------------|-----------|
| Стоимость расходов до РнС | `expenses_cost` | «Укажите фактические расходы заёмщика до получения РнС: проектирование, согласования, подготовка участка...» |
| Сумма бриджа | `credit_for_rights` | «Запрашиваемая сумма кредита — бридж или первый транш до получения РнС.» |
| Рыночная стоимость с РнС | `market_value_rights` | «Оценка рыночной стоимости прав на участок после получения РнС (ARV).» |

#### Сценарий Б: Покупка прав + РнС = ДА

| Поле | Системное имя | Подсказка |
|------|---------------|-----------|
| Стоимость покупки прав | `rights_price` | «Укажите стоимость приобретения прав на проект.» |
| Сумма кредита на покупку | `credit_for_rights` | «Запрашиваемая сумма кредита на покупку прав.» |
| Рыночная стоимость с РнС | `market_value_rights` | **Скрывается с UI**. Под капотом: `market_value_rights` = `rights_price` |

#### Сценарий В: Реализация собственного проекта + РнС = НЕТ

Те же три поля с теми же подсказками, что в **Сценарии А**.

## 8. Алгоритм расчёта

### Шаг 1. Открытие формы и автоопределение застройщика

При нажатии кнопки «Экспресс-расчёт» открывается форма. Система пытается автоматически определить группу застройщика по ЕПК ID → holding_id → «Нет в списке» (см. раздел 6.2).

### Шаг 2. Заполнение формы пользователем

Пользователь заполняет форму: Группа из реестра застройщиков (при необходимости), регион, параметры проекта, параметры сделки, LLCR от клиента (`llcr_value`), особенности сделки.

### Шаг 3. Расчет показателей по застройщику

После нажатия «Запустить анализ»:
- Если найден в `developers` → получает `category`, `tier_id`, `total_built_sqm`, `focus_regions`, `better_than_market`
- Если «Нет в списке» → tier 4, опыт «Отсутствует», `is_focus_region` = false

### Шаг 4. Определение опыта группы

```
ЕСЛИ total_built_sqm = 0 ИЛИ застройщик не найден
ТО `experience` = "Отсутствует"

ИНАЧЕ ЕСЛИ total_built_sqm < project_area_sqm
ТО `experience` = "Недостаточный"

ИНАЧЕ
ТО `experience` = "Достаточный"
```

Надбавка по опыту применяется только для не-ТОП (через `strategies`).

### Шаг 7. Проверка стоп-факторов

**Таблица:** `strategies`, тип «стоп фактор».

1. Регион = Москва И Тип недвижимости = Апартаменты → **Отказ**
2. Группа риска = «Стратегия не определяется» → **Отказ**

Если сработал: `traffic_light` = red, `decision` = decline, остальные шаги не выполняются.

### Шаг 8. Определение нормативных значений (`ltc_min`, `ltarv_min`, `equity_min`)

**8.1. Проверка индивидуальной стратегии**

Если группа совпадает со `group_name` в `strategies` (тип «стратегия») → берутся все нормативы из строки (`llcr_max`, `ltc_min`, `ltarv_min`, `equity_min`). Шаги 8.2 и 9 пропускаются.

**8.2. Нормативы**

**Таблица:** `strategies`, тип «нормативы». Применяется для всех продуктов.

Выбор строки по категории + `is_focus_region` + `is_high_potential`:
1. `is_focus_region` = true → строка с `is_focus_region` = true
2. Иначе `is_high_potential` = true → строка с `is_high_potential` = true
3. Иначе → строка, где оба = false

### Шаг 9. Расчёт нормативного LLCR (`llcr_max`)

Если индивидуальная стратегия не найдена:

**`llcr_max` = `llcr_base` + Σ(значения `llcr` из всех применимых строк «надбавка LLCR»)**

Где `llcr_base` = 1,20.

Система фильтрует все строки типа «надбавка LLCR», проверяет условия, суммирует положительные и отрицательные значения.

**Ограничения CAP и FLOOR (только для жилой недвижимости — не ИЖС, не апартаменты):**

| Условие | CAP (макс) | FLOOR (мин) |
|---------|-----------|-------------|
| Жилая **И** есть РнС | 1,35 | 1,15 |
| Жилая **И** нет РнС | 1,35 | 1,20 |

```
ЕСЛИ property_type = Жилая:
  llcr_uncapped = llcr_base + Σ надбавок
  ЕСЛИ llcr_uncapped > CAP:
    llcr_max = CAP; llcr_capped = true
  ИНАЧЕ ЕСЛИ llcr_uncapped < FLOOR:
    llcr_max = FLOOR; llcr_floored = true
  ИНАЧЕ
    llcr_max = llcr_uncapped
ИНАЧЕ (Апартаменты / ИЖС):
  llcr_max = llcr_uncapped; llcr_capped = false; llcr_floored = false
```

Где CAP и FLOOR зависят от `has_rns` (см. таблицу выше).

Признаки:
- `llcr_capped` = true, если LLCR был ограничен сверху (CAP)
- `llcr_floored` = true, если LLCR был ограничен снизу (FLOOR)
- `llcr_uncapped` — исходное значение до применения CAP/FLOOR

### Шаг 10. Расчёт метрик проекта

**LTC:**
```
ltc_value = credit_for_rights / база_LTC

База_LTC:
  has_rns = НЕТ: expenses_cost → rights_price → market_value_rights (первое непустое)
  has_rns = ДА: rights_price → market_value_rights (первое непустое)
```

**LTARV:**
```
ltarv_value = credit_for_rights / market_value_rights
```

**Equity:**
```
equity_value(%) = (1 − ltc_value) × 100%
```

### Шаг 11. Определение условий (жёсткие требования)

| Условие | Системное имя | Логика присвоения «да» |
|---------|---------------|----------------------|
| `LTC_BREACH` | `cond_ltc_breach` | `ltc_value` > `ltc_min` **И** `has_rns` = НЕТ |
| `LTARV_BREACH` | `cond_ltarv_breach` | `ltarv_value` > `ltarv_min` **И** `has_rns` = НЕТ |
| `LLCR_LOW` | `cond_llcr_low` | `llcr_value` < `llcr_max` |
| `NO_EXPERIENCE` | `cond_no_experience` | `experience` = «Отсутствует» |
| `LOW_EXPERIENCE` | `cond_low_experience` | `experience` = «Недостаточный» |

> **Примечание:** Условие `LOW_EQUITY` удалено. Показатель Equity (`equity_value` / `equity_min`) рассчитывается и выводится в отчёте. Нарушение собственного капитала отражается через `LTC_BREACH` и `LTARV_BREACH`.

### Шаг 14. Работа с LLM

**При стоп-факторе (красный):** LLM формирует формулировку отказа.

**При условиях (жёлтый):** LLM формирует пояснение нарушений и что делать.

**Для рисков:** LLM формирует для каждого риска (со значением «да») описание и меру минимизации на основе `description` и `mitigation` из таблицы `risks`.

### Шаг 15. Формирование отчёта

1. **Светофор и решение:** `traffic_light`, `decision`
2. **Условия** (жёсткие требования): `cond_ltc_breach`, `cond_ltarv_breach`, `cond_llcr_low`, `cond_no_experience`, `cond_low_experience` — да/нет
3. **Метрики:** `ltc_value` / `ltc_min`, `ltarv_value` / `ltarv_min`, `llcr_max` / `llcr_value` (с признаком `llcr_capped`), `equity_value` / `equity_min`
4. **Анализ рисков:** список рисков со значением «да», описание и меры минимизации (из LLM)
5. **Ставки финансирования:** диапазон ставок из `developers_tier`


### Шаг 16. Выявление рисков через LLM по `rm_notes`

#### Метод определения рисков

Система использует **гибридный подход** — два независимых канала:

| Канал | Источник | Метод | Коды |
|-------|----------|-------|------|
| Параметрический | Поля формы | Детерминированный (по условиям) | `NO_RNS`, `NO_TEP`, `APARTS`, `LONG_TERM`, `WEAK_DEMAND_PHASE` |
| Текстовый | Поле `rm_notes` | LLM → fallback по фразам | `INORGANIC_GROWTH`, `KRT_RISK`, `LAND_ZOUIT`, `LAND_SUBLEASE`, `NO_BENEFICIARY_SURETY_GROUP`, `NO_BENEFICIARY_SURETY_BRIDGE`, `OTHER_ACTIVITIES`, `MULTIPROJECTS`, `SHORT-TERM_LAND_LEASE`, `NO_LAND_LEASE`, `LOW_MARGIN`, `MARKETING_RISK` |

Результаты обоих каналов объединяются (дубликаты удаляются).

#### Параметрические риски (без LLM)

Определяются по полям формы — детерминированно, без вызова LLM:

| Код | Системное имя | Условие «да» |
|-----|---------------|-------------|
| `NO_RNS` | `risk_no_rns` | `has_rns` = НЕТ |
| `NO_TEP` | `risk_no_tep` | `has_teps` = НЕТ |
| `APARTS` | `risk_aparts` | `property_type` = Апартаменты |
| `LONG_TERM` | `risk_long_term` | `long_term_over_7y` = Да |
| `WEAK_DEMAND_PHASE` | `risk_weak_demand` | `risk_group` = «Высокая» |

#### Текстовые риски (через LLM)

##### Условие запуска

LLM вызывается **только при `len(rm_notes) ≥ 10` символов**. При пустом или коротком тексте текстовые риски не определяются.

##### Системный промт

```
РОЛЬ: Ты — эксперт банка по кредитному риску девелоперских проектов.
По свободному тексту заметок клиентского менеджера (RM) выделяешь уже
заведённые в системе коды риска (risk_code). Новые коды придумывать нельзя.

ЗАДАЧА: Прочитай текст rm_notes. Верни только те коды риска из
risk_code_arsenal, которые обоснованно следуют из текста.

ПРАВИЛА:
— Включай код только если факт явно описан в тексте rm_notes.
— Не добавляй код «из осторожности».
— Если из текста не следует ни один код — верни пустой массив risk_codes.
— Ориентируйся ТОЛЬКО на текст rm_notes; не восстанавливай факты паспорта
  сделки, если их нет в этом тексте.
— Каждый код — не более одного раза.
```

##### Арсенал кодов (`risk_code_arsenal`)

Передаётся в payload JSON. Содержит только коды, для которых есть описание и мера митигации в таблице `risks`:

| Код | Описание (передаётся LLM как `hint`) |
|-----|--------------------------------------|
| `INORGANIC_GROWTH` | Неорганический рост группы: слияния, поглощения, экспансия |
| `KRT_RISK` | Процедура корпоративного реструктурирования (КРТ) |
| `LAND_ZOUIT` | Земельный участок в зоне с особыми условиями использования (ЗОУИТ) |
| `LAND_SUBLEASE` | Земля в субаренде / переуступка прав аренды |
| `NO_BENEFICIARY_SURETY_GROUP` | Нет поручительства бенефициара (для ПФ / группы) |
| `NO_BENEFICIARY_SURETY_BRIDGE` | Нет поручительства бенефициара (для бриджа) |
| `OTHER_ACTIVITIES` | Прочая (неосновная) деятельность застройщика |
| `MULTIPROJECTS` | Многопроектность: распылённость на много объектов одновременно |
| `SHORT-TERM_LAND_LEASE` | Краткосрочная аренда земли (менее 5 лет до окончания) |
| `NO_LAND_LEASE` | Отсутствие оформленного права аренды / собственности на землю |
| `LOW_MARGIN` | Низкая маржинальность проекта |
| `MARKETING_RISK` | Маркетинговый риск: сложность продаж, низкий спрос, слабая ликвидность |

##### Формат ответа LLM

```json
{
  "risk_codes": ["INORGANIC_GROWTH", "LAND_SUBLEASE"],
  "rationale": "В тексте указано, что клиент растёт через слияния, а земля в субаренде."
}
```

Система валидирует ответ: оставляет только коды из `risk_code_arsenal`, приводит к верхнему регистру, удаляет дубликаты.

##### Эвристический fallback

Если LLM недоступен или вернул ошибку, система ищет ключевые фразы в нормализованном тексте `rm_notes`:

| Фрагмент (нормализованный) | Код риска |
|---------------------------|-----------|
| «неорганическ рост», «слияни», «поглощен» | `INORGANIC_GROWTH` |
| «крт» | `KRT_RISK` |
| «зоуит», «зона с особыми условиями» | `LAND_ZOUIT` |
| «субаренд», «аренд земл» | `LAND_SUBLEASE` |
| «нет поручител», «поручительства бенефициар» | `NO_BENEFICIARY_SURETY_GROUP` |
| «бенефициар» | `NO_BENEFICIARY_SURETY_BRIDGE` |
| «маржиналь», «низк марж», «сжат марж» | `LOW_MARGIN` |
| «ликвидн», «маркетингов риск», «сложн продаж» | `MARKETING_RISK` |
| «слаб спрос», «слабый спрос», «низк спрос» | `WEAK_DEMAND_PHASE` |
| «нет тэп», «без тэп», «отсутств тэп» | `NO_TEP` |
| «нет рнс», «без рнс», «отсутств рнс», «рнс» | `NO_RNS` |
| «апартамент» | `APARTS` |
| «долгосроч», «более 7 лет», «свыше 7 лет» | `LONG_TERM` |

> **Примечание:** полный словарь синонимов хранится в коде (`_RM_NOTES_CODE_HINTS`) и может расширяться без изменения БТ.

### 10.5 Техническая рекомендация: `focus_regions`

**Проблема:** колонка `focus_regions` заполнена только для 15 записей из 6921. Остальные записи содержат NULL.

**Рекомендация:** использовать пустой массив `[]` вместо NULL.

**Обоснование:**
- JSON-совместимость: `[]` — валидный JSON, NULL — нет (требует специальной обработки)
- Безопасность парсинга: `json.loads("[]")` всегда работает; `json.loads(None)` — ошибка
- Единообразие: все записи имеют одинаковый тип (array), упрощает SQL-запросы и ORM-маппинг
- Проверка `is_focus_region`: `region_name in focus_regions` работает одинаково для `[]` и NULL (оба дают false), но без guard-проверок на NULL

**Миграция:** при загрузке данных из Excel все NULL-значения в `focus_regions` заменяются на `[]`.
