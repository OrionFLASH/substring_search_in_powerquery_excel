## 1. Цель проекта

Автоматизировать привязку холдингов из `_HOLD_OD` к условным ГСЗ из `_base_gsz` в Excel 2016 без VBA — только Power Query (M).

**Результаты:**

1. Запрос `_HOLD_OD_GSZ` — `src/PowerQuery/_HOLD_OD_GSZ.pq`
2. Ускоренный запрос `_HOLD_OD_GSZ_FAST` — `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq`
3. Генератор справочника ключей — `src/generate_gsz_keys.py`
4. Параллельный Python-матчер — `src/gsz_matcher_parallel.py`
5. Локальный Excel `output/_base_gsz_keys.xlsx` (не в git)

## 2. Исходные данные

### 2.1 `_HOLD_OD` (вход)

| Колонка | Роль |
|---------|------|
| `Холдинг` | Текст поиска |
| `ID холдинга` | Сохраняется |
| `Sum([ОД текущий год])` | Сохраняется |
| `Sum([ОД прошлый год])` | Сохраняется |

**Выход запроса добавляет:** `условное ГСЗ`, `Отладка_совпадения_ГСЗ`.

### 2.2 `_base_gsz` (справочник)

| Колонка | Роль |
|---------|------|
| `Наименование, регион` | Значение при совпадении |
| `key_and_full_1` … `key_and_full_3` | AND, подстрока |
| `key_or_full_1` … `key_or_full_3` | OR, подстрока |
| `key_and_not_1` … `key_and_not_3` | AND, отдельное слово |
| `key_or_not_1` … `key_or_not_3` | OR, отдельное слово |

Пустые ячейки ключей игнорируются. Строка без ключей не участвует в сопоставлении.

### 2.3 `КейЛОАД.txt`

Список из ~6921 строки «Наименование, регион» для автоматической генерации ключей.

## 3. Бизнес-логика (согласовано)

### 3.1 Нормализация

`Text.Trim` + `Text.Lower` для «Холдинг» и всех ключей.

### 3.2 Режим `full`

Подстрока в любом месте текста.

### 3.3 Режим `not`

Отдельное слово: границы — начало/конец строки или не-буквенный символ. Буквы: кириллица + латиница.

### 3.4 AND

- Все непустые `key_and_full_*` и `key_and_not_*` обязательны.
- Позиции вхождений **не пересекаются** между всеми AND-полями строки.
- Если все AND пусты — блок считается пройденным.

### 3.5 OR

- Хотя бы одно непустое `key_or_full_*` или `key_or_not_*`.
- Если все OR пусты — блок не требуется.

### 3.6 AND + OR вместе

```
совпадение = AND_пройден И (OR_пуст ИЛИ OR_пройден)
```

### 3.7 Несколько совпадений

| Колонка | Правило |
|---------|---------|
| `условное ГСЗ` | Первое «Наименование, регион» по порядку в `_base_gsz` |
| `Отладка_совпадения_ГСЗ` | Все совпадения через `"; "` |
| Нет совпадений | `"-"` |

### 3.8 Прочее

- В ячейке ключа — одно слово.
- Новый запрос; `_HOLD_OD` не перезаписывается.
- Пустой «Холдинг» → `"-"`.

## 4. Архитектура Power Query

Файлы:
- `src/PowerQuery/_HOLD_OD_GSZ.pq` (базовая версия)
- `src/PowerQuery/_HOLD_OD_GSZ_FAST.pq` (ускоренная версия)

```
Источник_HOLD_OD ─────────────────────────────────────────┐
Источник_base_gsz → Справочник_буфер                      │
                                                          │
fnНормализоватьТекст                                      │
fnЭтоБуква                                                │
fnВсеПозицииFull / fnПозицииNot                           │
fnПроверитьAndТокены                                      │
fnПолеOrСовпало / fnПроверитьOrБлок                       │
fnСтрокаGszСовпала → fnНайтиВсеGsz                        │
                                                          ▼
                    СКолонкой_список_совпадений
                    СКолонкой_условное_ГСЗ
                    СКолонкой_отладка
                    Без_служебной_колонки → Результат
```

### Ограничения Excel 2016 (учтено)

| Проблема | Решение |
|----------|---------|
| `ExpandRecordColumn` 4 аргумента | Отдельные `Table.AddColumn` |
| `[_колонка]{0}` | `Record.Field` + `List.First` |
| `Table.AddColumn(..., type text)` | Без 4-го аргумента |
| `fnНайтиВсеGsz` 2 аргумента | Справочник в замыкании |
| `Table.Column` 3 аргумента | 2 аргумента + `List.Transform` |

## 5. Генератор ключей

Файл: `src/generate_gsz_keys.py`  
Конфиг: `config.json`

### Алгоритм

1. Чтение строк из `КейЛОАД.txt` (с 11-й строки).
2. Короткое имя = часть до первой запятой.
3. Токенизация: пробелы, дефисы, скобки; стоп-слова (ГК, Холдинг…).
4. Дефис → отдельные ключи (`ВКБ-Новостройки` → `ВКБ`, `Новостройки`).
5. Подбор 1–3 AND-токенов с проверкой уникальности (логика PQ).
6. Экспорт в `output/_base_gsz_keys.xlsx`.

### Статистика последнего прогона

| Метрика | Значение |
|---------|----------|
| Строк в списке | 6921 |
| С пометкой `ключи_задублированы` | ~5439 |
| Без токенов | ~152 |

Высокая доля дубликатов — из-за повторяющихся названий (ПИК, ФСК, Эталон, Самолет и др.). Строки с `да` требуют ручной доработки.

## 8. Производительность и оптимизация

Для рабочих книг с большим объемом данных (десятки тысяч строк `_HOLD_OD`) добавлен отдельный запрос `_HOLD_OD_GSZ_FAST` с сохранением бизнес-логики сопоставления:

1. Предподготовка строк `_base_gsz` (нормализация и токены) выполняется один раз.
2. Для каждой строки справочника выбирается "якорный" ключ для быстрого отсева кандидатов.
3. Полная проверка AND/OR запускается только для кандидатов после якорного фильтра.
4. В `full`-поиске используется `Text.PositionOf` с `Occurrence.All`.

Ожидаемый эффект: существенное ускорение на больших массивах при сохранении функционального поведения запроса.

## 9. Python Roadmap (с учетом требований по окружению)

Цель: полностью перенести логику сопоставления из M в Python с возможностью масштабирования по CPU и гибкой настройкой таблиц/колонок.

### Этап 1. Паритет логики (завершено)
- реализованы правила `full/not`, `and/or`, непересечение AND-токенов;
- сохранено поведение выходных колонок (`условное ГСЗ`, `Отладка_совпадения_ГСЗ`).

### Этап 2. Конфигурируемость источников (завершено)
- поддержан выбор таблиц: `_HOLD_OD`, `_base_gsz` (или пользовательские имена);
- поддержан выбор колонок и наборов key-колонок через CLI;
- добавлена поддержка JSON-конфига аргументов.

### Этап 3. Оптимизация скорости (завершено)
- предобработка `_base_gsz` в мета-структуру один раз;
- "якорный" предфильтр кандидатов;
- multiprocessing через `ProcessPoolExecutor`.
- индекс кандидатов по якорям (`not/full`) внутри воркеров;
- кэш позиций токенов на один холдинг для исключения повторных вычислений.

### Этап 4. Промышленная калибровка (следующий)
- замер времени на реальной книге (44k x 7k);
- подбор `workers` и `chunk-size` для конкретной машины;
- сверка результатов Python vs Power Query на контрольной выборке.
- рекомендуемый "живой" профиль логирования для длительных прогонов:
  - `work_batch_size = 20`
  - `progress_every_holdings = 500`
  - `progress_every_base_rows = 500`
  - `chunk_size = 100`

### Ограничения по зависимостям
- используется только Python + пакет `openpyxl` из стандартной поставки Anaconda;
- внешние нестандартные библиотеки не требуются.
